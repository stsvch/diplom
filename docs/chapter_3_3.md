# 3.3 Проектирование алгоритмов и описание ключевых модулей

Разработка надежного программного обеспечения требует тщательного проектирования логики обработки данных. На этапе проектирования алгоритмов определяются последовательности действий системы при выполнении ключевых сценариев, обрабатываются потенциальные исключительные ситуации и формируется архитектура взаимодействия между различными функциональными модулями.

В рамках разрабатываемого программного средства были выделены шесть базовых процессов, реализующих основные возможности системы: аутентификация пользователя с выдачей токенов доступа, автоматическая проверка тестовых заданий, фильтрация и постраничный вывод каталога курсов, загрузка файлов в объектное хранилище, обработка сдачи студенческих работ и доставка уведомлений в реальном времени. Для визуализации логики работы данных процессов спроектированы блок-схемы, выполненные в соответствии с требованиями ГОСТ 19.701-90, что позволяет однозначно интерпретировать последовательность серверных и клиентских операций.

## 3.3.1 Алгоритм аутентификации пользователя

Аутентификация реализована в модуле Auth и построена на связке JWT Access Token с коротким сроком жизни и Refresh Token, хранимого в httpOnly cookie. На рисунке 3.X представлена блок-схема алгоритма аутентификации пользователя.

*Рисунок 3.X -- Блок-схема алгоритма аутентификации пользователя*

Входными данными алгоритма являются адрес электронной почты и пароль, передаваемые клиентским приложением через POST-запрос к эндпоинту `/api/auth/login` в формате JSON.

Алгоритм начинает работу с поиска пользователя в базе данных PostgreSQL по указанному адресу электронной почты. Если запись не найдена, клиенту возвращается сообщение об ошибке «Неверный email или пароль» без уточнения, какой именно параметр указан неверно, что исключает перебор существующих учетных записей. Далее введенный пароль сверяется с хэшем, сохраненным в базе данных. Хэширование выполняется средствами ASP.NET Core Identity с использованием алгоритма PBKDF2. При несовпадении клиенту возвращается то же самое сообщение об ошибке. На следующем шаге проверяется, подтвержден ли адрес электронной почты. Если нет, вход отклоняется с предложением пройти подтверждение.

После прохождения всех трех проверок из базы данных загружаются роли пользователя (Student, Teacher, Admin). На их основе генерируется JWT Access Token со сроком действия 15 минут, содержащий идентификатор, имя, адрес электронной почты и список ролей. Параллельно создается Refresh Token -- случайная строка со сроком действия 7 дней, предназначенная для обновления сессии без повторного ввода пароля. Refresh Token записывается в базу данных с привязкой к пользователю и устанавливается в httpOnly cookie, недоступный из JavaScript, что предотвращает его перехват при XSS-атаках.

Выходными данными являются JWT Access Token и время его истечения, передаваемые в теле ответа, а также Refresh Token, установленный в cookie. При ошибке возвращается объект ApiError с описанием причины отказа.

## 3.3.2 Алгоритм автоматической проверки теста

Модуль тестирования отвечает за контроль знаний обучающихся. Алгоритм автоматической проверки запускается при завершении студентом попытки прохождения теста и выполняет оценку ответов, подсчет баллов и определение итогового статуса. На рисунке 3.X представлена блок-схема данного алгоритма.

*Рисунок 3.X -- Блок-схема алгоритма автоматической проверки теста*

Входными данными являются идентификатор попытки (attemptId) и идентификатор студента (studentId), передаваемые через POST-запрос к эндпоинту `/api/attempts/{id}/submit`.

На начальном этапе система загружает из базы данных попытку прохождения теста вместе со всеми вопросами, вариантами ответов и ответами студента. Если попытка не найдена, не принадлежит указанному студенту или уже была завершена ранее, возвращается ошибка. При успешной загрузке инициализируются переменные: totalScore = 0 (общий счет) и hasOpenAnswer = false (флаг наличия вопросов, требующих ручной проверки).

Далее для каждого ответа студента определяется тип вопроса и выполняется проверка. Для типа SingleChoice выбранный вариант сравнивается с правильным -- при совпадении начисляются баллы вопроса, иначе ноль. Для MultipleChoice множество выбранных вариантов сравнивается с множеством правильных; баллы начисляются только при полном совпадении. Для TextInput введенный текст сравнивается с эталонным ответом без учета регистра и пробелов по краям. Для Matching проверяется соответствие каждой составленной пары эталонному значению; баллы начисляются при полном совпадении всех пар. Для OpenAnswer ответ помечается как ожидающий ручной проверки преподавателем, а флаг hasOpenAnswer устанавливается в true.

После обработки всех ответов вычисляется итоговый балл суммированием. Статус попытки определяется по флагу hasOpenAnswer: если среди вопросов есть хотя бы один с открытым ответом, статус устанавливается в NeedsReview; в противном случае -- Completed. Результат записывается в базу данных с фиксацией балла, статуса и времени завершения. Клиенту возвращается детализированный результат с итоговым баллом, статусом и информацией по каждому вопросу.

## 3.3.3 Алгоритм фильтрации и пагинации каталога курсов

Каталог курсов является основным средством навигации для обучающихся и гостей системы. Алгоритм реализует динамическую фильтрацию, сортировку и постраничный вывод с использованием паттерна Specification, позволяющего формировать запрос к базе данных на основе набора критериев. На рисунке 3.X представлена блок-схема данного алгоритма.

*Рисунок 3.X -- Блок-схема алгоритма фильтрации и пагинации каталога курсов*

Входными данными являются параметры строки URL: идентификатор дисциплины (disciplineId), признак бесплатности (isFree), уровень сложности (level), поисковая строка (search), критерий сортировки (sortBy), номер страницы (page) и размер страницы (pageSize).

Алгоритм начинается с формирования базового фильтра, исключающего неопубликованные и архивированные курсы (IsPublished = true, IsArchived = false). Затем последовательно проверяется наличие каждого необязательного параметра. Если указан disciplineId, к спецификации добавляется условие отбора по дисциплине. Если задан isFree, отбираются бесплатные или платные курсы. Если указан level, применяется фильтр по уровню сложности. Если присутствует search, отбираются курсы, название или описание которых содержит указанную подстроку без учета регистра.

После применения всех фильтров подсчитывается общее количество подходящих записей (totalCount), необходимое для формирования информации о пагинации. Далее выполняется сортировка: по цене (возрастание), по популярности (убывание по количеству записавшихся), по дате создания (убывание) или по умолчанию -- по дате создания. Из отсортированной выборки пропускается (page - 1) * pageSize записей и отбирается pageSize записей. Полученные сущности преобразуются в объекты CourseListDto средствами AutoMapper. Клиенту возвращается объект PagedResult, содержащий список курсов, общее количество записей, номер страницы, размер страницы и общее число страниц.

## 3.3.4 Алгоритм загрузки и хранения файлов

Загрузка файлов используется в нескольких модулях системы: при размещении учебных материалов, прикреплении ответов к заданиям, вложениях в сообщениях. Алгоритм принимает файл от клиента, проверяет его, сохраняет в S3-совместимое объектное хранилище MinIO и регистрирует метаданные в PostgreSQL. На рисунке 3.X представлена блок-схема данного алгоритма.

*Рисунок 3.X -- Блок-схема алгоритма загрузки и хранения файлов*

Входными данными являются загружаемый файл (multipart/form-data), тип сущности-владельца (entityType) и идентификатор сущности (entityId).

Из имени файла извлекается расширение. Если оно входит в список запрещенных (.exe, .bat, .sh, .cmd, .ps1, .msi), загрузка отклоняется с сообщением об ошибке. Далее определяется допустимый максимальный размер: для видеофайлов (MIME-типы video/*) предел составляет 1 ГБ, для всех остальных -- 100 МБ. При превышении лимита загрузка отклоняется. Выполняется парсинг типа сущности-владельца (LessonBlock, Assignment, AssignmentSubmission и др.); при некорректном значении возвращается ошибка.

После прохождения валидации создается сущность Attachment с уникальным идентификатором, именем файла, типом содержимого, размером и привязкой к сущности-владельцу. Генерируется путь хранения в формате `{guid}_{fileName}`, исключающий конфликты имен. Файл загружается в MinIO в бакет eduplatform-files операцией PutObjectAsync. В сущности Attachment обновляются поля StoragePath и FileUrl. Метаданные сохраняются в таблицу Attachment базы данных PostgreSQL. Клиенту возвращается объект AttachmentDto с идентификатором, именем файла, URL для скачивания и размером.

## 3.3.5 Алгоритм обработки сдачи задания

Сдача заданий -- частый сценарий взаимодействия обучающегося с системой. Алгоритм принимает ответ студента и выполняет серию проверок, направленных на соблюдение ограничений, установленных преподавателем. На рисунке 3.X представлена блок-схема данного алгоритма.

*Рисунок 3.X -- Блок-схема алгоритма обработки сдачи задания*

Входными данными являются идентификатор задания (assignmentId), идентификатор студента (studentId) и текстовое содержание ответа (content), передаваемые через POST-запрос к эндпоинту `/api/assignments/{id}/submit`.

Система выполняет поиск задания в базе данных. Если запись не найдена, возвращается ошибка. Затем проверяется дедлайн: если он установлен и текущее время его превысило, сдача отклоняется с сообщением «Дедлайн сдачи истек». Далее загружаются предыдущие попытки сдачи данного задания текущим студентом. Если у студента уже есть работа в статусе «Отправлена» или «На проверке», повторная сдача отклоняется, поскольку одновременно допускается не более одной работы на рассмотрении. Если для задания установлено ограничение MaxAttempts, проверяется количество использованных попыток; при исчерпании лимита сдача отклоняется.

При прохождении всех проверок создается сущность AssignmentSubmission с номером попытки (AttemptNumber = количество предыдущих + 1), статусом Submitted и текущей датой. Сущность сохраняется в базу данных. Клиенту возвращается объект SubmissionDto с идентификатором, статусом, номером попытки и временем отправки. При ошибке возвращается ApiError с описанием причины отказа.

## 3.3.6 Алгоритм доставки уведомлений в реальном времени

Система уведомлений информирует пользователей о событиях: выставление оценки, приближение дедлайна, публикация нового задания, создание занятия. Алгоритм реализует двойной механизм доставки -- сохранение уведомления в базу данных для последующего отображения в интерфейсе и мгновенную push-доставку через WebSocket с использованием SignalR. На рисунке 3.X представлена блок-схема данного алгоритма.

*Рисунок 3.X -- Блок-схема алгоритма доставки уведомлений в реальном времени*

Алгоритм состоит из двух взаимосвязанных процессов.

Первый процесс -- подключение клиента. После аутентификации клиентское приложение устанавливает WebSocket-соединение с NotificationHub, передавая JWT-токен в качестве параметра авторизации. Hub проверяет валидность токена, извлекает идентификатор пользователя из Claims и добавляет текущий ConnectionId в SignalR-группу, соответствующую этому идентификатору. Клиент подписывается на событие «ReceiveNotification» и переходит в состояние ожидания. Если токен невалиден, соединение отклоняется.

Второй процесс -- формирование и доставка уведомления. При возникновении события в системе (например, при выставлении оценки за задание) соответствующий модуль создает сущность Notification с идентификатором получателя, типом уведомления, заголовком, текстом сообщения и ссылкой на связанный объект. Признак прочтения (IsRead) устанавливается в false. Сущность сохраняется в таблицу Notification базы данных PostgreSQL. Далее она преобразуется в NotificationDto и через IHubContext отправляется в SignalR-группу получателя методом «ReceiveNotification».

Если пользователь подключен к NotificationHub, уведомление мгновенно доставляется на клиент, где отображается в виде всплывающего сообщения и обновляется счетчик непрочитанных уведомлений в заголовке. Если пользователь не подключен, уведомление остается в базе данных и будет доступно при следующем обращении к эндпоинту `/api/notifications` или при повторном подключении. Такой подход гарантирует, что уведомления не теряются при отсутствии активного соединения, а при его наличии доставляются без задержки.
