# 4 Создание программного средства

На этапе создания программного средства выполняется практическая реализация технических решений, принятых в предыдущих разделах, а также сборка и отладка системы как единого программного комплекса. Для проекта EduPlatform данный этап включает разработку серверной и клиентской частей, настройку хранилищ данных и вспомогательных сервисов, реализацию программных интерфейсов взаимодействия между модулями, а также проверку работоспособности основных пользовательских сценариев.

Разрабатываемое программное средство представляет собой веб-приложение для организации учебного процесса, включающее инструменты управления курсами, учебным контентом, тестированием, заданиями, расписанием, сообщениями и уведомлениями. С учетом многофункциональности системы и необходимости разделения ответственности между отдельными частями приложения при реализации был использован стек технологий, обеспечивающий модульность, расширяемость, удобство сопровождения и возможность независимой разработки клиентской и серверной частей.

В отличие от главы 3, где рассматривались архитектурные и логические проектные решения, в данной главе основной акцент сделан на уровне программной реализации. Ниже описываются конкретные модули, классы, интерфейсы, методы, механизмы обмена данными, а также порядок сборки и результаты отладки разработанного программного средства.

## 4.1 Обоснование выбора средств и технологий реализации

Окончательный выбор средств реализации программного средства был выполнен с учетом функциональных и нефункциональных требований, сформулированных на этапе проектирования. При выборе технологий учитывались следующие критерии: возможность построения современной клиент-серверной архитектуры, поддержка модульной разработки, наличие развитых средств работы с базами данных, удобство реализации REST API и механизмов взаимодействия в реальном времени, а также доступность инструментов сборки, развертывания и сопровождения.

Для реализации серверной части выбран язык программирования C# и платформа ASP.NET Core. Данный стек предоставляет развитую инфраструктуру для создания веб-приложений, поддерживает внедрение зависимостей, маршрутизацию, аутентификацию и авторизацию, работу с промежуточным программным обеспечением, а также хорошо подходит для реализации модульной архитектуры и паттерна CQRS. Дополнительным преимуществом является тесная интеграция с Entity Framework Core, что упрощает доступ к реляционным данным и поддержку миграций схемы базы данных.

Клиентская часть реализована на языке TypeScript с использованием фреймворка Angular. Выбор Angular обусловлен наличием встроенной маршрутизации, развитой системы компонентной архитектуры, поддержки реактивных форм, механизмов внедрения зависимостей и удобных средств организации крупного SPA-приложения. Использование TypeScript позволяет повысить надежность клиентского кода за счет статической типизации, а также упростить сопровождение и развитие большого количества экранов, сервисов и моделей данных.

В качестве основной реляционной базы данных используется PostgreSQL. Данная система управления базами данных применяется для хранения пользователей, курсов, уроков, тестов, заданий, оценок, прогресса, уведомлений, календарных событий и расписания. Выбор PostgreSQL обусловлен надежностью, поддержкой транзакций, развитой системой индексации, поддержкой схем, а также возможностью хранения полуструктурированных данных в формате JSONB, что особенно важно для реализации блочной системы контента уроков.

Для хранения чатов и сообщений используется MongoDB. Применение документо-ориентированной базы данных в данном случае связано с особенностями структуры сообщений и диалогов, а также с необходимостью гибкой работы с документами переменной структуры. Такое разделение хранилищ позволяет использовать сильные стороны каждой технологии: PostgreSQL - для строго структурированных предметных данных, MongoDB - для коллекций сообщений и чатов.

Для хранения пользовательских файлов, учебных материалов и вложений применяется объектное хранилище MinIO, совместимое с S3 API. Использование отдельного файлового хранилища позволяет не перегружать реляционную базу данных бинарными данными, централизовать доступ к файлам и обеспечить унифицированный механизм загрузки, хранения и скачивания вложений в разных модулях системы.

Для реализации обмена данными между клиентской и серверной частями используется REST API. Такой подход обеспечивает четкое разделение ответственности между frontend и backend, позволяет использовать стандартные HTTP-методы для выполнения операций чтения и изменения данных, а также упрощает тестирование и документирование прикладных интерфейсов. Для сценариев, требующих оперативной доставки данных пользователю, например при получении уведомлений и обмене сообщениями, дополнительно применяется технология SignalR, обеспечивающая взаимодействие в режиме, близком к реальному времени.

Сборка и запуск компонентов системы осуществляются с использованием нескольких инструментов. Для поднятия инфраструктурных сервисов, включающих PostgreSQL, MongoDB и MinIO, используется Docker Compose. Серверная часть запускается через .NET Aspire AppHost, что позволяет централизованно управлять backend-приложением в процессе разработки. Клиентская часть запускается средствами Angular CLI. Такое разделение упрощает локальную разработку, так как инфраструктура поднимается в контейнерах, а прикладной код backend и frontend остается удобным для отладки и модификации.

## 4.2 Реализация архитектуры серверной части программного средства

На уровне программной реализации серверная часть EduPlatform построена в виде модульного монолита. Такой подход позволяет объединить все функциональные подсистемы в одном серверном приложении, сохранив при этом четкое логическое разделение предметных областей. В отличие от традиционного монолита, где бизнес-логика часто сосредоточена в ограниченном наборе слоев без явного разграничения ответственности, модульный монолит организует код по независимым функциональным модулям, каждый из которых инкапсулирует собственные сущности, прикладную логику и инфраструктурные зависимости.

Структура серверной части проекта включает четыре верхнеуровневые области: `Host`, `Modules`, `Shared` и `AppHost`. Компонент `Host` представляет собой основную точку входа серверного приложения и отвечает за конфигурацию HTTP-конвейера, подключение middleware, маршрутизацию, настройку Swagger, CORS, аутентификации, авторизации и регистрацию SignalR-хабов. Каталог `Modules` содержит прикладные функциональные модули системы. Каталог `Shared` используется для размещения общих моделей, базовых классов, контрактов и инфраструктурных компонентов, применяемых несколькими модулями. Компонент `AppHost` используется для управления запуском серверной части в среде разработки через .NET Aspire.

На уровне логической организации backend разделен на следующие функциональные модули: `Auth`, `Courses`, `Content`, `Tests`, `Assignments`, `Grading`, `Progress`, `Notifications`, `Calendar`, `Messaging`, `Scheduling`. Такое разбиение соответствует основным подсистемам предметной области образовательной платформы и позволяет изолировать правила обработки данных, сущности и прикладные сервисы внутри каждого модуля.

Внутри каждого функционального модуля используется единый принцип послойного построения, основанный на подходе Onion Architecture. Как правило, модуль включает проекты `Domain`, `Application` и `Infrastructure`. Слой `Domain` содержит сущности предметной области, перечисления и value object-ы. Слой `Application` реализует прикладную логику, включая команды и запросы, их обработчики, DTO, интерфейсы доступа к данным, маппинги и правила валидации. Слой `Infrastructure` отвечает за работу с внешними ресурсами: контекстами баз данных, файловым хранилищем, механизмами отправки уведомлений, SignalR-хабами и другими интеграциями.

Слой `Host` не содержит бизнес-логики предметной области. Его назначение состоит в предоставлении внешнего программного интерфейса в виде REST API и маршрутизации запросов в прикладной слой. Контроллеры `Host` принимают HTTP-запросы, выполняют извлечение параметров, проверяют права доступа, после чего передают обработку в команды и запросы MediatR. Например, контроллер каталога курсов формирует объект `GetCourseCatalogQuery` и передает его в обработчик прикладного слоя, который выполняет построение выборки, фильтрацию, пагинацию и преобразование данных в DTO. Аналогичный подход используется и в других контроллерах, включая работу с уроками, блоками контента, тестами, заданиями, расписанием и уведомлениями.

Дополнительным элементом унификации серверной части является выделение общего слоя `Shared`. В нем размещены элементы, используемые несколькими модулями одновременно: базовый тип результата выполнения операций `Result<T>`, модели ошибок API, общие перечисления, pipeline-behavior для централизованной валидации запросов и набор контрактов межмодульного взаимодействия. Такой подход позволяет уменьшить связанность между модулями и избежать прямой зависимости одной предметной области от внутренней реализации другой. Вместо этого используются абстракции, например сервисы чтения и вспомогательные контракты, через которые модули получают доступ к необходимым данным.

Связь между слоями серверной части организована последовательно. Контроллеры `Host` инициируют выполнение сценариев, передавая управление в слой `Application`. Прикладной слой опирается на интерфейсы доступа к данным и сервисам, не завися от конкретного способа их реализации. Реальные контексты баз данных, файловые сервисы, хабы и другие адаптеры предоставляются в слое `Infrastructure` через механизм внедрения зависимостей. Благодаря этому достигается слабая связанность между логикой приложения и техническими деталями хранения и передачи данных.

Важной особенностью реализации является централизованная регистрация модулей в точке входа приложения. При запуске `Host` в контейнер служб последовательно подключаются все функциональные модули системы, а затем выполняется настройка кросс-модульных сервисов: глобальной валидации запросов, CORS, логирования, обработки исключений, аутентификации и SignalR. Кроме того, при старте приложения выполняется применение миграций для контекстов баз данных и начальное заполнение отдельных справочных данных, что упрощает подготовку среды разработки и гарантирует согласованность структуры хранилищ.

## 4.3 Реализация архитектуры клиентской части программного средства

Клиентская часть программного средства EduPlatform реализована как одностраничное веб-приложение на базе Angular. Такой подход обеспечивает быстрый отклик пользовательского интерфейса, повторное использование компонентов и удобную организацию большого количества экранов в рамках единого приложения. При программной реализации frontend основное внимание уделялось разделению ответственности между уровнями интерфейса, изоляции бизнес-сценариев по функциональным разделам и построению удобного механизма навигации для пользователей с различными ролями.

Архитектурно клиентская часть разделена на четыре основные области: `core`, `shared`, `features` и `layouts`. Каталог `core` содержит базовые сервисы, модели, guards и interceptors, необходимые для функционирования всего приложения. В нем размещены сервисы аутентификации, работы с сайдбаром, интеграции с SignalR и загрузки файлов, а также инфраструктурные элементы, связанные с контролем доступа и централизованной обработкой HTTP-запросов. Каталог `shared` включает переиспользуемые компоненты интерфейса, директивы, pipes и утилиты, применяемые в различных функциональных разделах. Каталог `features` содержит реализацию прикладных пользовательских сценариев, сгруппированных по предметным разделам системы. Каталог `layouts` отвечает за каркас отображения страниц, включая гостевой интерфейс, прикладной layout, заголовок и боковую навигацию.

Для организации frontend используется современная конфигурация Angular-приложения на базе standalone-компонентов. Регистрация маршрутов выполняется через `provideRouter`, а конфигурация HTTP-клиента - через `provideHttpClient` с подключением цепочки interceptors. Такой подход уменьшает связанность между частями клиентского приложения и упрощает подключение новых экранов и компонентов. Дополнительно в конфигурации приложения подключен редактор кода Monaco, используемый в сценариях, связанных с заданиями по программированию и блоками типа `CodeExercise`.

Навигация в клиентском приложении построена на основе ролевой модели. На верхнем уровне маршрутизации выделены четыре группы путей: гостевая часть, а также разделы `student`, `teacher` и `admin`. Гостевая часть содержит маршруты входа, регистрации, подтверждения электронной почты и восстановления пароля. Разделы авторизованных пользователей защищаются с помощью `authGuard` и `roleGuard`, что позволяет, с одной стороны, запрещать доступ неавторизованным пользователям, а с другой - перенаправлять пользователя в допустимую для его роли область системы. Такое решение обеспечивает централизованный контроль доступа на уровне маршрутов и позволяет избежать дублирования проверок внутри экранов.

Для авторизованных пользователей используется общий прикладной layout, содержащий верхнюю панель и боковое меню навигации. Состав пунктов бокового меню формируется динамически в зависимости от роли текущего пользователя. Для обучающегося отображаются разделы каталога курсов, календаря, сообщений, уведомлений, оплаты и профиля. Для преподавателя доступны разделы управления курсами, проверкой работ, журналом оценок, расписанием, сообщениями и уведомлениями. Для администратора предусмотрены разделы управления пользователями, курсами, дисциплинами, аналитикой и настройками. Такой подход делает интерфейс адаптивным к роли пользователя и уменьшает визуальную перегрузку.

Функциональные возможности клиентской части сгруппированы по feature-разделам. В проекте выделены разделы `auth`, `courses`, `content`, `tests`, `assignments`, `grading`, `calendar`, `notifications`, `messaging`, `scheduling`, `profile`, `progress`, `admin`. Такое разбиение соответствует общей структуре серверной части и упрощает согласование клиентской и серверной логики. Например, раздел `courses` содержит страницы каталога, карточки курса, редактор курса, редактор урока и экран просмотра урока, раздел `tests` реализует создание, прохождение и проверку тестов, а раздел `assignments` содержит средства создания и сдачи заданий.

Сервисный слой клиентской части организован централизованно. Базовым сервисом является `AuthService`, который отвечает за вход пользователя, регистрацию, восстановление сессии, получение профиля, обновление токена и выход из системы. Состояние текущего пользователя и токена доступа хранится в реактивных сущностях Angular, что позволяет автоматически обновлять элементы интерфейса при изменении состояния авторизации. Для автоматического прикрепления токена к HTTP-запросам используется `authInterceptor`, а для унифицированного преобразования ответов с ошибками - `errorInterceptor`. В результате компоненты прикладного уровня получают уже нормализованные данные и освобождаются от необходимости повторять типовые механизмы работы с авторизацией и ошибками.

Существенную роль в построении клиентской части играет каталог `shared`, содержащий переиспользуемые элементы пользовательского интерфейса. В нем размещены компоненты карточек курсов, файлов, кнопок, полей ввода, прогресс-баров, редакторов и просмотрщиков rich-text, загрузчиков файлов, toast-уведомлений, видеоплеера и других элементов. Использование общей библиотеки компонентов обеспечивает единообразие интерфейса, снижает объем дублируемого кода и упрощает разработку новых экранов.

## 4.4 Разработка диаграммы классов и описание ключевых классов модуля Content

В качестве ключевого функционального модуля в рамках главы 4 целесообразно рассмотреть модуль `Content`, отвечающий за конструирование, хранение и прохождение учебного содержимого уроков. Выбор данного модуля обусловлен тем, что он содержит наиболее характерные для проекта программные решения: типизированную модель предметной области, полиморфное хранение структур данных, набор интерфейсов расширения, автоматическую и ручную проверку ответов, а также тесную интеграцию с клиентской частью, где преподаватель формирует урок из набора блоков, а обучающийся проходит его в интерактивном режиме. Именно на примере этого модуля удобно раскрыть требование методических указаний о построении диаграммы классов и подробном описании атрибутов, методов и связей между классами.

Концептуально модуль `Content` реализует представление урока как упорядоченной последовательности блоков различного типа. В отличие от традиционной модели, где урок хранится как единый текстовый материал, в данной системе он рассматривается как контейнер, включающий информационные, интерактивные и составные элементы. Это позволяет комбинировать в одном уроке текстовые пояснения, видео, изображения, задания с автопроверкой, упражнения по программированию, встроенные тесты и встроенные задания. Такой подход повышает гибкость формирования учебного материала и позволяет адаптировать урок под конкретную дисциплину и формат обучения.

Основу доменной модели модуля составляют классы `LessonBlock` и `LessonBlockAttempt`. Класс `LessonBlock` описывает отдельный элемент урока и содержит идентификатор урока `LessonId`, позицию в последовательности `OrderIndex`, тип блока `Type`, данные блока `Data`, общие настройки `Settings`, а также временные метки создания и обновления. Класс `LessonBlockAttempt` предназначен для хранения результата взаимодействия обучающегося с конкретным блоком. В нем фиксируются идентификатор блока и пользователя, отправленный ответ `Answers`, набранный балл, максимальный балл, признак правильности, признак необходимости ручной проверки, количество использованных попыток, статус проверки, а также сведения о проверяющем преподавателе и комментарии к проверке. Таким образом, один класс описывает учебную единицу, а второй - факт ее прохождения конкретным пользователем.

Существенной особенностью реализации является использование абстрактного базового класса `LessonBlockData`, представляющего типизированные данные блока. Для него настроена полиморфная сериализация, позволяющая хранить в одном поле данные различных конкретных типов. В текущей реализации поддерживаются восемнадцать типов блоков, разделенных на четыре группы: информационные (`Text`, `Video`, `Audio`, `Image`, `Banner`, `File`), автоматически проверяемые (`SingleChoice`, `MultipleChoice`, `TrueFalse`, `FillGap`, `Dropdown`, `WordBank`, `Reorder`, `Matching`), блоки с ручной или полуавтоматической проверкой (`OpenText`, `CodeExercise`), а также составные блоки (`Quiz`, `Assignment`). Например, класс `TextBlockData` содержит HTML-представление текстового содержания, класс `CodeExerciseBlockData` хранит формулировку задания, язык программирования, стартовый код, тестовые примеры и ограничения на выполнение, а `QuizBlockData` содержит идентификатор связанного теста.

Общие настройки поведения блока вынесены в отдельный класс `LessonBlockSettings`. В нем задаются параметры, одинаково значимые для разных типов блоков: количество баллов `Points`, признак обязательности для завершения урока `RequiredForCompletion`, подсказка `Hint`, признак отображения обратной связи `ShowFeedback`, ограничение по числу попыток `MaxAttempts` и дополнительные настройки, связанные с перемешиванием вариантов. Выделение такого класса позволяет не дублировать общие параметры в каждой структуре данных блока и обеспечивает единообразную обработку результатов прохождения урока.

Для обеспечения расширяемости модуля применен набор интерфейсов, отвечающих за валидацию и проверку блоков. Корректность структуры данных при создании и изменении блока обеспечивается интерфейсом `IBlockDataValidator`. Для каждого типа блока может быть определен собственный валидатор, а выбор нужной реализации выполняется через `BlockDataValidatorRegistry`. В текущей реализации зарегистрированы валидаторы для всех восемнадцати типов блоков, что позволяет проверять согласованность входных данных до сохранения в базу данных. Проверка ответов обучающегося реализуется через интерфейс `IBlockGrader`, который определяет метод `Grade(...)` и возвращает объект `GradeResult` с баллом, максимальным баллом, признаком правильности, признаком необходимости ручной проверки и текстовой обратной связью. Диспетчеризацию вызова по типу блока выполняет `BlockGraderRegistry`. Для автоматически проверяемых и частично проверяемых интерактивных блоков зарегистрировано десять реализаций проверщиков.

Практическая реализация ключевых пользовательских сценариев выполнена в прикладном слое через команды, запросы и их обработчики. Создание нового блока осуществляется обработчиком `CreateLessonBlockCommandHandler`. На первом этапе он проверяет совпадение типа блока и типа переданных данных, затем передает структуру данных в реестр валидаторов. При успешной проверке определяется максимальный индекс порядка среди уже существующих блоков урока, после чего новый блок сохраняется как следующий элемент последовательности. Таким образом, обработчик инкапсулирует как бизнес-правило согласования типа и содержимого блока, так и правило автоматического включения блока в упорядоченную структуру урока.

Одним из наиболее важных сценариев модуля является отправка ответа обучающимся и вычисление результата. Эта логика реализована в `SubmitAttemptCommandHandler`. Обработчик загружает блок по идентификатору, проверяет соответствие типа ответа типу блока, контролирует ограничение на число попыток, после чего через `BlockGraderRegistry` вызывает соответствующий проверщик. Если попытка отсутствует, создается новая запись `LessonBlockAttempt`; если она уже существует, данные попытки обновляются, а счетчик использованных попыток увеличивается. После сохранения результата обработчик дополнительно рассчитывает, выполнены ли все обязательные блоки урока, и при положительном результате вызывает межмодульный контракт `ILessonProgressUpdater`, который помечает урок как завершенный в модуле `Progress`. Это решение демонстрирует, как модуль `Content` взаимодействует с другими частями системы без жесткой зависимости от их внутренней реализации.

Для блоков, которые требуют участия преподавателя, реализован сценарий ручной проверки. Обработчик `ReviewAttemptCommandHandler` позволяет выставить балл в допустимом диапазоне, снять признак `NeedsReview`, изменить статус попытки на `Graded`, зафиксировать идентификатор проверяющего, комментарий и время проверки. Благодаря этому модуль поддерживает как полностью автоматизированные сценарии контроля знаний, так и режим ручного оценивания развернутых текстовых ответов.

Специальный подтип логики реализован для блока `CodeExercise`. Его данные представлены классом `CodeExerciseBlockData`, который содержит язык программирования, стартовый код, набор тестовых случаев, лимит времени выполнения и ограничение по памяти. На прикладном уровне выполнение кода реализуется обработчиком `ExecuteCodeCommandHandler`, который извлекает настройки конкретного блока и передает их в интерфейс `ICodeExecutor`. В текущей реализации используется сервис `ProcessCodeExecutor`, выполняющий запуск кода через дочерний процесс. Такой подход достаточен для этапа прототипирования и демонстрации работы модуля, однако в комментариях к реализации зафиксировано, что для промышленного развертывания необходима изолированная sandbox-среда.

Инфраструктурный уровень модуля представлен интерфейсом `IContentDbContext` и классом `ContentDbContext`. Контекст содержит наборы данных `LessonBlocks`, `LessonBlockAttempts` и `Attachments`. При конфигурации моделей используется схема `content`, а поля `Data`, `Settings` и `Answers` сохраняются в PostgreSQL как `jsonb` с пользовательскими преобразованиями сериализации и десериализации. Такое решение позволяет совместить строгую реляционную организацию основных сущностей с гибкостью хранения полиморфных структур данных. Для обеспечения целостности данных заданы индексы по идентификатору урока и порядку блока, а также уникальный индекс по паре `(BlockId, UserId)`, гарантирующий единственность текущей попытки пользователя для каждого блока.

В клиентской части модуль `Content` представлен сервисами `ContentService` и `BlockAttemptsService`, а также набором специализированных компонентов редактирования и просмотра блоков. Сервис `ContentService` инкапсулирует запросы на получение списка блоков урока, создание, изменение, удаление и перестановку блоков, а также выполнение кода для упражнения `CodeExercise`. Сервис `BlockAttemptsService` отвечает за отправку ответов, получение текущей попытки, загрузку прогресса по уроку, получение попыток для проверки и выполнение ручной проверки. Такое разделение делает клиентский код более понятным: один сервис связан с конструированием содержимого урока, второй - с прохождением и оцениванием.

Отображение и редактирование конкретного типа блока реализуется через компоненты-хосты `BlockEditorHostComponent` и `BlockViewerHostComponent`. Эти компоненты принимают объект блока и в зависимости от его типа динамически подключают специализированный компонент редактирования или просмотра. В редакторе преподавателя дополнительно используется `BlockInserterComponent` и меню типов блоков `BlockTypeMenuComponent`, где все типы сгруппированы по четырем категориям: контент, упражнения с автопроверкой, ручная проверка и вставка составных элементов. На уровне экранов интеграция модуля реализована в `LessonEditorComponent`, который обеспечивает автосохранение изменений, дублирование блоков, перестановку drag-and-drop и применение шаблонов уроков, а также в `LessonViewComponent`, который загружает блоки, связанные попытки обучающегося, обновляет прогресс по уроку и отображает обратную связь после отправки ответа.

## 4.5 Описание интерфейсов методов и функций модулей

При реализации программного средства существенное внимание было уделено описанию интерфейсов методов, функций и контрактов взаимодействия как между внутренними модулями backend, так и между клиентской и серверной частями системы. Наличие четко определенных программных интерфейсов позволяет снизить связанность компонентов, упростить сопровождение системы и обеспечить расширяемость прикладной логики без изменения уже реализованных модулей.

Внутренние связи серверной части построены на основе контрактов, вынесенных в общий слой `EduPlatform.Shared.Application.Contracts`. Такой подход позволяет модулям обращаться друг к другу не напрямую, а через абстракции. Например, модуль `Courses` может получать сведения о тестах и заданиях через интерфейсы `ITestReadService` и `IAssignmentReadService`, модуль `Content` сообщает о завершении урока через интерфейс `ILessonProgressUpdater`, модуль `Messaging` использует `INotificationDispatcher` для формирования уведомлений о новых сообщениях, а модуль `Calendar` предоставляет контракт `ICalendarEventPublisher` для создания и обновления событий, связанных с дедлайнами и расписанием. Аналогично, интерфейс `IEnrollmentReadService` обеспечивает доступ к сведениям о действующих записях на курс, а `IChatAdmin` используется для администрирования курсовых чатов.

Подобная организация интерфейсов особенно важна в модульном монолите, поскольку позволяет сохранить границы предметных областей внутри одного исполняемого приложения. Каждый модуль реализует собственные сервисы доступа к данным и бизнес-правила, но при необходимости предоставляет наружу только ограниченный набор операций чтения, публикации событий или административных действий. За счет этого можно изменять внутреннюю реализацию модуля, не затрагивая остальные части системы, если внешний контракт остается неизменным.

Помимо внутренних контрактов, на уровне серверной части широко используются прикладные интерфейсы в виде команд, запросов и DTO. Контроллеры слоя `Host` принимают HTTP-запрос, преобразуют входные данные в объект команды или запроса, после чего передают его в `MediatR`. Результат выполнения возвращается в виде DTO или объекта ошибки API. Такой подход унифицирует работу всех контроллеров и позволяет описывать прикладные интерфейсы единообразно: входные параметры задаются объектом запроса, выходные данные - DTO, а логика обработки инкапсулируется в handler-е соответствующего модуля.

Внешние прикладные интерфейсы системы реализованы двумя основными способами. Первый способ - REST API, используемый для большинства операций создания, чтения, обновления и удаления данных. Второй способ - взаимодействие в режиме, близком к реальному времени, реализованное через SignalR-хабы. На стороне клиента REST API используется сервисами Angular, работающими через `HttpClient`, тогда как SignalR применяется для доставки уведомлений и обмена сообщениями без необходимости постоянного опроса сервера.

Наиболее показательными внешними интерфейсами системы являются методы аутентификации, работы с курсами, уроками, заданиями, тестами, уведомлениями и расписанием. Так, модуль `Auth` предоставляет методы регистрации, входа, обновления токена и восстановления пароля; модуль `Courses` реализует каталог курсов, получение детальной информации о курсе, публикацию и запись на курс; модуль `Content` предоставляет методы получения блоков урока, изменения структуры урока, отправки ответов и запуска кода; модуль `Assignments` отвечает за создание и сдачу заданий; модуль `Tests` - за создание, прохождение и проверку тестов; модуль `Notifications` - за получение списка уведомлений и управление их статусом; модуль `Scheduling` - за работу со слотами занятий и бронированиями.

## 4.6 Отладка отдельных модулей и устранение ошибок

После определения архитектуры, структуры модулей и программных интерфейсов была выполнена непосредственная реализация серверной и клиентской частей системы, а затем организована поэтапная отладка отдельных модулей. Такой порядок работы позволил сначала сформировать общую основу приложения, включающую конфигурацию backend, клиентскую маршрутизацию, механизмы аутентификации, доступ к базам данных и файловому хранилищу, а затем последовательно проверять функциональные подсистемы на уровне отдельных сценариев.

На стороне backend программирование осуществлялось в рамках модульной структуры с разделением на слои `Domain`, `Application` и `Infrastructure`. В первую очередь были реализованы базовые элементы общей инфраструктуры: шаблон результата выполнения операций `Result<T>`, глобальная обработка исключений, централизованная валидация запросов через `ValidationBehavior`, конфигурация контроллеров, аутентификация и авторизация, подключение контекстов баз данных и механизмы применения миграций при старте приложения. После этого были последовательно реализованы прикладные модули: управление курсами и уроками, блочная система контента, тестирование, задания, оценки, прогресс, уведомления, сообщения и расписание.

На стороне frontend программирование строилось вокруг feature-ориентированной структуры Angular-приложения. Сначала были настроены базовые маршруты, guards, interceptors, сервис аутентификации и каркасные layout-компоненты. Далее реализовывались отдельные функциональные разделы: формы входа и регистрации, каталог курсов, просмотр и редактирование уроков, прохождение тестов, сдача заданий, уведомления, обмен сообщениями и расписание. Такой подход позволил локально проверять каждый реализуемый экран и сервис без необходимости завершать весь проект целиком.

Важным элементом процесса разработки стала поэтапная локальная отладка отдельных модулей. Для серверной части она выполнялась с использованием Swagger UI, логирования и встроенного механизма обработки ошибок. Все необработанные исключения перехватываются `ExceptionHandlingMiddleware` и преобразуются в единый формат `ApiError`, что облегчает поиск причин сбоев при вызове прикладных методов. Ошибки валидации запросов централизованно обрабатываются через `ValidationBehavior`, поэтому некорректные входные данные выявляются до выполнения бизнес-логики. На стороне frontend ответы с ошибками приводятся к единому формату через `errorInterceptor`, а затем отображаются пользователю с помощью `ToastService`. В результате процесс отладки охватывает сразу обе стороны системы: от backend-обработчика до визуального представления результата в интерфейсе.

В ходе программирования особое внимание уделялось проверке корректности обмена данными между клиентской и серверной частями. Для этого выполнялась отладка DTO, маршрутов API, правил сериализации и десериализации, а также согласованности перечислений и структур данных в C# и TypeScript. Наиболее характерным примером является модуль `Content`, где серверная часть использует полиморфные типы `LessonBlockData` и `LessonBlockAnswer`, а клиентская часть - discriminated union-модели для представления данных и ответов блоков. Отладка этого модуля включала проверку создания блоков, сохранения настроек, отправки попыток, расчета результатов и отображения обратной связи в пользовательском интерфейсе.

Для модулей, использующих интеграцию с внешними или вспомогательными сервисами, выполнялась дополнительная проверка инфраструктурного поведения. Так, для модуля файлов реализована проверка загрузки вложений в MinIO, для модуля уведомлений - доставка событий через `NotificationHub`, для модуля сообщений - работа `ChatHub`, а для модуля расписания - сохранение и получение слотов и бронирований через PostgreSQL. Отдельно выполнялась отладка прикладного сценария `CodeExercise`, включающего отправку исходного кода, запуск тестовых примеров и возврат результата выполнения на клиент.

Практическая отладка проводилась в основном по сценарному принципу. Для каждого ключевого модуля определялся минимальный набор пользовательских действий, подтверждающий его работоспособность. Например, для `Auth` таким сценарием являлись регистрация, вход и получение профиля пользователя; для `Courses` - создание курса, получение списка курсов и запись обучающегося; для `Content` - создание урока с блоками и прохождение этих блоков; для `Tests` - запуск попытки и завершение теста; для `Assignments` - создание задания, сдача работы и выставление оценки; для `Notifications` и `Messaging` - получение событий в реальном времени; для `Scheduling` - создание слота и запись на занятие. Такой подход позволил проверять не отдельные строки кода, а законченные фрагменты прикладного поведения.

В процессе локальной отладки выявлялись и устранялись типовые для веб-приложений ошибки: некорректные параметры запросов, нарушения ограничений бизнес-логики, ошибки привязки маршрутов, рассогласование форматов данных между backend и frontend, проблемы сериализации комплексных структур, ошибки прав доступа и некорректное отображение результатов в интерфейсе. Существенную роль в стабилизации модулей сыграла унификация обработки исключений и ответов об ошибках, благодаря чему диагностическая информация стала доступна как на сервере, так и на клиенте.

## 4.7 Сборка проекта и инструкция по запуску программного средства

Завершающим этапом создания программного средства является сборка всех его компонентов и подготовка инструкции по их корректному запуску. Для проекта EduPlatform данный этап включает подготовку инфраструктуры, запуск серверной и клиентской частей, а также выполнение контрольной сборки backend и frontend.

Сборка и запуск системы организованы по комбинированной схеме. Инфраструктурные сервисы, необходимые для работы приложения, поднимаются через Docker Compose. В их состав входят PostgreSQL как основная реляционная база данных, MongoDB для хранения чатов и сообщений, а также MinIO как объектное хранилище файлов. Такое решение позволяет изолировать внешние зависимости приложения от прикладного кода и быстро подготавливать среду для разработки и проверки системы.

После запуска инфраструктуры выполняется старт серверной части. Backend запускается через `.NET Aspire` из проекта `AppHost`, который подключает серверное приложение как проектный ресурс. При этом серверная часть использует строки подключения к PostgreSQL, MongoDB и MinIO, заданные в конфигурации приложения. На этапе запуска выполняется не только инициализация HTTP-конвейера, контроллеров, хабов и middleware, но и применение миграций ко всем используемым контекстам данных, а также начальное заполнение отдельных данных, включая роли пользователей и справочник дисциплин. Это позволяет получать согласованное состояние базы данных без выполнения дополнительных ручных операций.

Следующим этапом является запуск клиентской части. Frontend запускается средствами Angular CLI. В среде разработки обращения к серверной части выполняются через proxy-конфигурацию, перенаправляющую маршруты `/api` и `/hubs` на backend по адресу `http://localhost:5000`. Такой механизм позволяет не изменять клиентский код при локальном запуске и обеспечивает корректную работу как HTTP-запросов, так и WebSocket-соединений для SignalR.

Для подтверждения работоспособности проекта выполняется не только запуск приложений, но и их сборка. Серверная часть успешно собирается командой `dotnet build backend\\EduPlatform.sln`, а клиентская часть - командой `npm run build` в каталоге `frontend`. В ходе проверки подтверждено успешное завершение сборки обоих компонентов без критических ошибок. Для backend были зафиксированы предупреждения о версиях отдельных внешних библиотек, однако они не препятствуют компиляции и запуску приложения. На стороне frontend результатом сборки является сформированный production-бандл Angular-приложения.

Отдельную роль в данном подразделе играет последовательность запуска компонентов. Если инфраструктурные сервисы недоступны, backend не сможет корректно применить миграции и обработать запросы, а при недоступности backend клиентское приложение не сможет загрузить данные и установить SignalR-соединения. Поэтому инструкция сборки и запуска должна задавать строгий порядок действий: сначала инфраструктура, затем серверная часть, затем клиентская часть. Такой порядок обеспечивает готовность всех зависимостей к моменту начала работы пользовательского интерфейса.

## 4.8 Комплексная отладка и подтверждение работоспособности системы

После завершения локальной отладки отдельных модулей была выполнена комплексная проверка программного средства на уровне сквозных пользовательских сценариев. В отличие от модульной отладки, где анализируется корректность работы отдельных обработчиков, сервисов и экранов, на данном этапе проверяется согласованность взаимодействия нескольких подсистем в рамках единого прикладного процесса.

Для обучающегося контролировалась следующая последовательность действий: аутентификация в системе, просмотр каталога курсов, открытие карточки курса, запись на курс, переход к уроку, прохождение учебных блоков, отправка попыток, получение результатов и обновление прогресса. Для преподавателя проверялись сценарии создания курса, редактирования структуры урока, добавления и изменения блоков контента, публикации заданий и тестов, просмотра отправленных работ и выставления оценок. Для подсистем реального времени дополнительно проверялись доставка уведомлений и обмен сообщениями, а для модуля расписания - создание слотов, запись обучающегося и отображение бронирований.

Комплексная отладка также подтверждает корректность интеграции клиентской и серверной частей с используемыми хранилищами данных и вспомогательными сервисами. При проведении проверки учитывалось, что backend должен иметь доступ к PostgreSQL, MongoDB и MinIO, а frontend - к REST API и SignalR-хабам. Только при совместной работоспособности всех перечисленных компонентов возможно выполнение полных пользовательских сценариев без нарушения логики приложения.

Дополнительным подтверждением работоспособности системы являются результаты контрольной сборки. Серверная часть успешно компилируется командой `dotnet build backend\\EduPlatform.sln`, а клиентская часть - командой `npm run build` в каталоге `frontend`. Успешное завершение указанных операций подтверждает корректность зависимостей, отсутствие критических ошибок компиляции и возможность получения исполняемых артефактов для дальнейшего запуска и демонстрации системы.

Таким образом, комплексная отладка подтвердила, что разработанное программное средство функционирует как единый программный комплекс. Реализованные модули взаимодействуют корректно, инструкция по запуску обеспечивает воспроизводимость развертывания, а результаты сборки и сквозной проверки подтверждают достижение цели этапа создания программного средства.
