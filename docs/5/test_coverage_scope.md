# Перечень покрытия тестами EduPlatform

Документ определяет, **что именно** в разработанном программном средстве подлежит проверке тестами. Сформирован на основе фактически реализованных модулей backend и feature-папок frontend (а не на основе абстрактных «требований к LMS»). Документ является основой для:

- технического задания на разработку тестов;
- итогового набора тест-кейсов;
- подраздела 5.3 «Разработка тестов для проверки работоспособности программного средства» (см. `chapter_5_plan.md`).

---

## 1. Принципы покрытия

Покрытие строится одновременно по трём осям, чтобы тесты убедительно подтверждали соответствие ТЗ:

1. **Покрытие требований** — каждое ключевое функциональное требование закрыто хотя бы одним сценарием.
2. **Покрытие сквозных пользовательских сценариев** — основные действия проходят от начала до конца (создание курса → наполнение → прохождение студентом → выставление оценки).
3. **Покрытие ролей** — каждая из трёх ролей (`Admin`, `Teacher`, `Student`) проверена в её типовых операциях и в сценариях запрета доступа к чужим функциям.

Каждый пункт ниже снабжён приоритетом:

- **P1** — критический: без него систему нельзя считать работоспособной;
- **P2** — высокий: важно для полноты функциональности;
- **P3** — средний: при наличии времени.

| Интерфейс, адаптивность, real-time (UI, L, M) | 10 % |

## 4. Что **не** входит в покрытие

| Область | Причина исключения |
|---|---|
| Автоматизированные unit/integration/e2e-тесты в коде | CLAUDE.md явно запрещает писать тесты на текущем этапе |
| Нагрузочное тестирование (производительность, конкурентность) | Не входит в задачи дипломной работы |
| Тестирование безопасности уровня pen-test (SQL-injection, XSS) | Базовая защита обеспечивается ASP.NET Core/EF; глубокий аудит вне рамок |
| Кроссбраузерная матрица | Достаточно одной актуальной версии Chromium-браузера |
| Содержимое frontend-feature `placeholder/` | Это заведомо незавершённые экраны-заглушки |
| Внутренние слои Onion (Domain/Application/Infrastructure) по отдельности | Тестируется только наблюдаемое поведение через API и UI |
| Stripe webhooks в production-режиме | Достаточно проверки через test-keys и Stripe CLI |
| Содержание `AdminStatsController`, `PlatformSettingsController`, `ReportsController` сверх минимального smoke-теста | Точный набор endpoints требует доуточнения, не относится к ядру ТЗ |

## 6. Критерии достаточности покрытия

Тестирование признаётся достаточным, если одновременно выполнены условия:

1. Все пункты с приоритетом **P1 пройдены** без блокирующих дефектов.
2. Покрыто **не менее 80 % пунктов P2**.
3. Проверены **все три роли** (`Admin`, `Teacher`, `Student`) — как в позитивных, так и в негативных сценариях разграничения доступа.
4. Пройдены **все сценарии зоны R** с приоритетом P1.
5. Каждая из 17 функциональных зон содержит **минимум один пройденный позитивный и один негативный** сценарий.
6. По каждому из **18 типов блоков контента** выполнен хотя бы один сценарий создания и просмотра.
7. Все выявленные критические дефекты устранены и подтверждены повторной проверкой.
