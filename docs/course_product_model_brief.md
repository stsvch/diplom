# Верхнеуровневая модель курса для обсуждения продукта и дизайна

Дата: 2026-04-26

## Назначение документа

Этот документ нужен для обсуждения с командой, дизайнерами и другими участниками проекта.

Цель - не описать техническую реализацию до уровня таблиц и миграций, а зафиксировать, как должен восприниматься курс в продукте:

- из каких частей состоит курс;
- как эти части связаны между собой;
- какие сценарии должен проходить преподаватель;
- какие сценарии должен проходить студент;
- какие зоны нужно продумать в дизайне;
- какие вопросы нужно решить до глубокой переработки.

Технические детали БД здесь указаны только на уровне "что примерно нужно хранить".

## Главная идея

Курс должен быть не просто набором уроков.

Курс - это верхнеуровневый учебный продукт, внутри которого могут быть:

- уроки;
- тесты;
- задания;
- live-занятия;
- расписание;
- файлы и дополнительные материалы;
- глоссарий;
- прогресс студентов;
- оценки;
- сертификаты;
- рейтинг;
- отзывы;
- комментарии и обсуждения.

Сейчас в системе курс больше похож на структуру:

```text
Курс -> Модуль -> Урок -> Блоки урока
```

Но продуктово курс должен восприниматься шире:

```text
Курс
  -> Раздел / модуль
    -> Урок
    -> Тест
    -> Задание
    -> Live-занятие
    -> Материал
```

То есть внутри курса должны быть разные учебные элементы, а не только уроки.

## Что такое курс

Курс - это центральная сущность платформы.

Он объединяет:

- информацию для каталога;
- программу обучения;
- учебные материалы;
- проверочные активности;
- расписание занятий;
- студентов;
- прогресс;
- оценки;
- оплату и доступ;
- обратную связь.

Для преподавателя курс - это рабочее пространство, где он создает и управляет обучением.

Для студента курс - это понятный маршрут прохождения: что смотреть, что читать, что выполнить, когда прийти на занятие, что уже завершено и что осталось.

## Основные части курса

### 1. Информация о курсе

Это верхний слой курса, который виден в каталоге и на странице курса.

Нужно хранить и показывать:

- название курса;
- краткое описание;
- подробное описание;
- обложку;
- дисциплину;
- уровень сложности;
- теги;
- преподавателя;
- цену или бесплатность;
- длительность;
- количество элементов курса;
- рейтинг;
- количество отзывов;
- статус: черновик, опубликован, архивирован.

Для дизайна важно:

- страница курса должна сразу объяснять, что это за курс и зачем его проходить;
- преподаватель должен понимать, какие поля влияют на каталог;
- студент должен быстро понять пользу курса, формат и требования.

### 2. Структура курса

Структура курса - это программа обучения.

Она должна отвечать на вопрос:

```text
В каком порядке студент проходит курс?
```

Структура может состоять из разделов или модулей.

Внутри разделов находятся элементы курса:

- урок;
- тест;
- задание;
- live-занятие;
- файл или дополнительный материал;
- внешний ресурс.

Пример:

```text
Курс "Основы веб-разработки"

Раздел 1. Введение
  1. Урок: Что такое веб
  2. Урок: Как работает браузер
  3. Тест: Проверка по введению

Раздел 2. HTML и CSS
  1. Урок: HTML-разметка
  2. Практическое задание: Сверстать страницу
  3. Live-занятие: Разбор работ

Раздел 3. Итог
  1. Итоговый тест
  2. Итоговое задание
```

Для дизайна важно:

- преподаватель должен легко собирать структуру;
- элементы разных типов должны быть визуально отличимы;
- должно быть понятно, что является обязательным, а что дополнительным;
- нужно показывать, какие элементы готовы, а какие еще в черновике.

### 3. Урок

Урок - это один учебный элемент внутри курса.

Урок нужен для подачи материала.

Внутри урока могут быть блоки:

- текст;
- видео;
- аудио;
- изображение;
- файл;
- баннер;
- небольшое упражнение;
- открытый ответ;
- кодовое упражнение.

Урок может быть:

- одной страницей;
- пошаговым прохождением;
- теоретическим;
- практическим;
- смешанным.

Важно не смешивать понятия:

- урок - это контейнер для учебного материала;
- блок - это маленькая часть урока;
- тест и задание могут быть отдельными элементами курса, а не обязательно блоками внутри урока.

Для дизайна важно:

- редактор урока должен быть простым;
- библиотека блоков должна быть сгруппирована по смыслу;
- у каждого блока должен быть статус готовности;
- пустые блоки должны быть нормальным состоянием черновика, а не ошибкой.

### 4. Тест

Тест - это проверочная активность.

Он может быть:

- отдельным элементом курса;
- итогом раздела;
- итогом всего курса;
- встроенным в урок, если это нужно для короткой проверки.

Тест содержит:

- название;
- описание;
- вопросы;
- варианты ответов;
- настройки попыток;
- ограничение по времени;
- дедлайн;
- настройки показа правильных ответов;
- баллы.

Для дизайна важно:

- преподаватель должен иметь возможность создать тест прямо в контексте курса;
- не должно быть ощущения, что тест создается "где-то отдельно";
- если тест добавляется после урока, пользователь должен понимать, к какому разделу он относится;
- вопросы теста должны редактироваться в удобном конструкторе.

### 5. Задание

Задание - это активность, где студент отправляет работу, а преподаватель проверяет.

Задание может быть:

- отдельным элементом курса;
- практикой после урока;
- итоговым проектом;
- частью модуля.

Задание содержит:

- название;
- описание;
- критерии оценивания;
- дедлайн;
- максимальный балл;
- количество попыток;
- отправки студентов;
- комментарии преподавателя;
- статус проверки.

Для дизайна важно:

- задание должно создаваться прямо из курса или урока;
- студент должен четко видеть, что нужно сделать;
- преподаватель должен видеть очередь работ на проверку;
- критерии оценивания должны быть заметны и понятны.

### 6. Live-занятия и расписание

Курс может включать занятия по расписанию.

Есть два возможных формата:

- разовое занятие;
- регулярное занятие по правилу.

Примеры:

- каждую среду в 18:00;
- консультация перед итоговой работой;
- групповое занятие по модулю;
- индивидуальное занятие.

Live-занятие может быть элементом курса, как урок или тест.

Оно должно иметь:

- название;
- дату и время;
- длительность;
- ссылку на встречу;
- формат: индивидуальное или групповое;
- лимит участников;
- связь с курсом;
- связь с календарем.

Для дизайна важно:

- расписание должно быть видно и преподавателю, и студенту;
- live-занятие должно отображаться в структуре курса, если оно важно для прохождения;
- календарь должен быть связан с курсом, а не жить отдельно;
- студент должен понимать, нужно ли записываться на занятие.

### 7. Материалы и ресурсы

Курс может содержать дополнительные материалы:

- файлы;
- ссылки;
- презентации;
- изображения;
- PDF;
- глоссарий;
- список литературы;
- внешние ресурсы.

Материалы могут быть:

- внутри урока;
- отдельным элементом курса;
- общими ресурсами курса.

Для дизайна важно:

- нужно решить, где студент видит все материалы курса;
- нужно отличать обязательные материалы от дополнительных;
- нужно дать преподавателю простой способ прикреплять файлы и ссылки.

### 8. Участники и доступ

У курса есть участники.

Основные роли:

- преподаватель;
- студент;
- администратор.

Студент может получить доступ к курсу через:

- запись на бесплатный курс;
- покупку курса;
- подписку;
- ручное добавление администратором или преподавателем в будущем.

Для дизайна важно:

- студент должен понимать, есть ли у него доступ;
- если курс платный, путь покупки должен быть понятным;
- преподаватель должен видеть количество студентов;
- администратор должен видеть состояние курса и доступов.

### 9. Прогресс

Прогресс показывает, как студент проходит курс.

Прогресс должен учитывать не только уроки, но и другие элементы:

- урок просмотрен или завершен;
- тест пройден;
- задание отправлено или проверено;
- live-занятие посещено;
- обязательные материалы открыты;
- итоговый курс завершен.

Для дизайна важно:

- студент должен видеть общий прогресс курса;
- студент должен видеть, что делать дальше;
- преподаватель должен видеть прогресс группы;
- элементы курса должны показывать состояние: не начато, в процессе, завершено, на проверке, просрочено.

### 10. Оценки

Оценки могут появляться из разных источников:

- тест;
- задание;
- ручная оценка преподавателя;
- упражнение внутри урока;
- итоговая оценка за курс.

Важно:

- оценка должна быть связана с курсом;
- оценка должна иметь источник;
- студент должен видеть свои результаты;
- преподаватель должен видеть журнал оценок.

Для дизайна важно:

- нужен понятный gradebook;
- студенту нужен простой экран "Мои результаты";
- преподавателю нужен фильтр по курсу, студенту, заданию, тесту.

### 11. Сертификат

Сертификат - это результат завершения курса.

Он может выдаваться, если:

- курс пройден полностью;
- выполнены обязательные элементы;
- итоговый балл выше минимального;
- все задания проверены;
- студент оплатил курс, если курс платный.

Для дизайна важно:

- студент должен видеть условия получения сертификата;
- преподаватель должен понимать, включен ли сертификат;
- нужно показывать прогресс до сертификата.

### 12. Рейтинг и отзывы

Курс может иметь рейтинг и отзывы.

Рейтинг отвечает на вопрос:

```text
Как студенты оценили курс?
```

Отзыв содержит:

- оценку от 1 до 5;
- текст;
- автора;
- дату;
- статус модерации.

Для дизайна важно:

- рейтинг должен быть виден в каталоге;
- отзывы должны быть видны на странице курса;
- студент должен понимать, когда он может оставить отзыв;
- преподаватель должен видеть обратную связь;
- администратор должен иметь возможность скрывать неподходящие отзывы.

### 13. Комментарии и обсуждения

Комментарии отличаются от отзывов.

Отзыв - это публичная оценка курса.

Комментарий или обсуждение - это коммуникация во время обучения.

Комментарии могут быть:

- к курсу;
- к уроку;
- к заданию;
- к конкретному вопросу студента.

Это можно делать позже, но при проектировании модели курса стоит оставить место для обсуждений.

Для дизайна важно:

- не смешивать отзывы и учебные обсуждения;
- обсуждения должны помогать учиться, а не перегружать страницу курса;
- преподаватель должен видеть новые вопросы.

## Как все связано

Упрощенная карта:

```text
Курс
  содержит информацию для каталога
  содержит структуру обучения
  содержит разделы
  содержит элементы курса
  имеет студентов
  имеет расписание
  имеет прогресс
  имеет оценки
  имеет отзывы

Раздел
  принадлежит курсу
  группирует элементы курса

Элемент курса
  принадлежит курсу
  может находиться в разделе
  может быть уроком, тестом, заданием, live-занятием или ресурсом
  имеет порядок
  имеет статус готовности
  может быть обязательным или дополнительным

Урок
  является элементом курса
  содержит блоки

Тест
  является элементом курса или частью урока
  содержит вопросы
  имеет попытки студентов

Задание
  является элементом курса или частью урока
  имеет отправки студентов
  проверяется преподавателем

Live-занятие
  является элементом курса или событием расписания
  отображается в календаре

Студент
  получает доступ к курсу
  проходит элементы курса
  получает прогресс и оценки
  может оставить отзыв
```

## Как это должно выглядеть для преподавателя

Преподавателю нужен единый конструктор курса.

Идеальный сценарий:

1. Преподаватель нажимает "Создать курс".
2. Выбирает пустой курс или шаблон.
3. Заполняет основную информацию.
4. Собирает структуру курса.
5. Добавляет уроки, тесты, задания и live-занятия.
6. Наполняет уроки блоками.
7. Настраивает дедлайны, обязательность и баллы.
8. Смотрит чеклист готовности.
9. Открывает предпросмотр.
10. Публикует курс.

Конструктор курса должен показывать:

- что уже готово;
- что еще не заполнено;
- какие элементы обязательные;
- какие элементы будут видны студенту;
- какие ошибки мешают публикации.

## Как это должно выглядеть для студента

Студенту нужен понятный маршрут прохождения.

Идеальный сценарий:

1. Студент открывает страницу курса.
2. Видит описание, преподавателя, рейтинг, программу и цену.
3. Записывается или покупает курс.
4. Попадает на страницу прохождения.
5. Видит следующий рекомендуемый элемент.
6. Проходит уроки.
7. Выполняет тесты и задания.
8. Видит расписание live-занятий.
9. Следит за прогрессом.
10. Получает оценки и обратную связь.
11. После завершения получает сертификат.
12. Может оставить отзыв.

Студент не должен думать, где отдельно тесты, где отдельно задания, где отдельно календарь.

Для него это должно быть единым маршрутом внутри курса.

## Что примерно нужно хранить

Без глубоких технических деталей, системе нужно хранить:

- данные курса;
- разделы курса;
- элементы курса;
- тип элемента курса;
- порядок элементов;
- обязательность элементов;
- статус готовности элементов;
- уроки и блоки уроков;
- тесты и вопросы;
- задания и отправки;
- live-занятия и расписание;
- записи студентов на курс;
- прогресс по элементам;
- оценки;
- покупки и доступы;
- отзывы и рейтинг;
- комментарии и обсуждения в будущем.

Важно: технически это может храниться в разных модулях, но для пользователя должно выглядеть как единый курс.

## Что нужно продумать дизайнерам

### 1. Страница создания курса

Нужно решить:

- показывать ли сначала выбор шаблона;
- сколько шагов должно быть в мастере;
- какие поля обязательные;
- где объяснить модель курса.

### 2. Конструктор курса

Нужно продумать:

- как выглядит дерево курса;
- как добавляются уроки, тесты, задания, live-занятия;
- как различаются типы элементов;
- как показывается статус готовности;
- где находится чеклист публикации;
- как работает предпросмотр.

### 3. Редактор урока

Нужно продумать:

- как добавлять блоки;
- как сгруппировать блоки;
- как показать пустой блок;
- как показать ошибку заполнения;
- как объяснить, чем блок отличается от теста или задания.

### 4. Редактор теста

Нужно продумать:

- как создать тест внутри курса;
- как добавлять вопросы;
- как показать настройки теста;
- как показать связь теста с курсом.

### 5. Редактор задания

Нужно продумать:

- как создать задание внутри курса;
- как описать критерии оценивания;
- как настроить дедлайн и баллы;
- как показать отправки студентов.

### 6. Расписание

Нужно продумать:

- как добавить live-занятие в курс;
- как показать разовые и регулярные занятия;
- как связать расписание с календарем;
- как студент записывается на занятие.

### 7. Страница прохождения курса

Нужно продумать:

- как студент видит программу;
- как видит текущий шаг;
- как переходит между элементами;
- как видит прогресс;
- как видит дедлайны;
- как видит оценки.

### 8. Рейтинг и отзывы

Нужно продумать:

- где рейтинг показывается в каталоге;
- где отзывы показываются на странице курса;
- когда студент может оставить отзыв;
- как выглядит форма отзыва;
- как преподаватель видит отзывы.

## Важные продуктовые вопросы

Эти вопросы нужно обсудить до финального дизайна.

### Структура курса

- Обязательны ли модули или курс может быть просто списком элементов?
- Может ли тест быть отдельным элементом курса без урока?
- Может ли задание быть отдельным элементом курса без урока?
- Может ли live-занятие быть обязательным для завершения курса?
- Можно ли создавать материалы без урока?

### Прохождение

- Должен ли студент проходить курс строго по порядку?
- Может ли преподаватель смешивать свободное и последовательное прохождение?
- Что считается завершением урока?
- Что считается завершением теста?
- Что считается завершением задания?
- Как учитывать live-занятия в прогрессе?

### Проверка и оценки

- Все ли тесты влияют на итоговый балл?
- Все ли задания влияют на итоговый балл?
- Нужен ли минимальный балл для завершения курса?
- Можно ли проходить курс без оценивания?

### Расписание

- Расписание является обязательной частью курса или дополнительной?
- Нужны ли регулярные занятия?
- Нужно ли студенту записываться на каждое занятие?
- Что делать, если студент пропустил занятие?

### Отзывы и рейтинг

- Когда студент может оставить отзыв?
- Можно ли оставить отзыв без завершения курса?
- Можно ли редактировать отзыв?
- Должны ли отзывы проходить модерацию?
- Видит ли преподаватель автора отзыва?

### Сертификат

- Когда выдавать сертификат?
- Нужен ли минимальный процент прохождения?
- Нужен ли минимальный балл?
- Что делать, если часть заданий еще не проверена?

## Что важно не потерять при переработке

Нужно сохранить:

- текущие курсы;
- текущие уроки;
- текущие блоки уроков;
- текущие тесты;
- текущие задания;
- текущие отправки заданий;
- текущие попытки тестов;
- текущий прогресс студентов;
- текущие оценки;
- текущие платежи и доступы;
- текущие календарные события.

Переработка должна быть постепенной.

Сначала нужно сделать новую понятную модель отображения курса, а потом аккуратно переводить старые сценарии на нее.

## Рекомендуемый подход к проектированию

Не начинать с деталей таблиц.

Сначала нужно согласовать:

1. Как пользователь понимает курс.
2. Какие элементы могут быть внутри курса.
3. Как преподаватель собирает курс.
4. Как студент проходит курс.
5. Как система показывает готовность курса.
6. Как тесты, задания и расписание входят в структуру курса.
7. Где находятся отзывы, рейтинг и обсуждения.

После этого можно уточнять техническую модель.

## Короткая формулировка для команды

Нужно спроектировать курс как единое учебное пространство.

Курс должен включать не только уроки, но и тесты, задания, live-занятия, материалы, прогресс, оценки, расписание, рейтинг и отзывы.

Для преподавателя это должен быть единый конструктор курса.

Для студента это должен быть единый маршрут прохождения.

Для системы это должна быть модель, где разные элементы курса связаны между собой и могут участвовать в публикации, прогрессе, оценивании и сертификате.
