# Проблемы UX создания и наполнения курса

Дата фиксации: 2026-04-26

## Контекст

Сейчас процесс создания курса разделен на несколько независимых экранов:

1. Выбор шаблона курса: `/teacher/courses/new`.
2. Создание/редактирование основных данных курса: `/teacher/courses/create`.
3. Редактор структуры курса: `/teacher/courses/:id/editor`.
4. Редактор наполнения урока: `/teacher/lesson/:id/edit`.
5. Отдельный редактор теста: `/teacher/test/new` или `/teacher/test/:id/edit`.
6. Отдельный редактор задания: `/teacher/assignment/new` или `/teacher/assignment/:id/edit`.

Базовая модель системы: **курс -> модуль -> урок -> блок**.

Проблема в том, что эта модель не объясняется пользователю до начала работы. Поэтому преподаватель видит несколько разных редакторов и не понимает, где именно создается структура, где наполняется урок, а где создаются проверочные материалы.

## Основные проблемы

### 1. Нет единого понятного сценария создания курса

Сейчас создание курса выглядит как переход между разными страницами, а не как единый процесс.

Пользователь сначала заполняет данные курса, затем попадает в редактор структуры, затем отдельно открывает каждый урок, затем отдельно создает тесты и задания.

Почему это плохо:

- Непонятно, какой следующий шаг после сохранения курса.
- Непонятно, что сначала нужно создать модуль, потом урок, потом блоки.
- Пользователь может попытаться опубликовать курс до того, как он реально наполнен.
- Нет ощущения прогресса: курс создан, но непонятно, готов ли он.

Приоритет: высокий.

### 2. Разные входы в создание курса

В сайдбаре пункт "Создать курс" ведет на `/teacher/courses/new`, где можно выбрать шаблон.

В списке "Мои курсы" кнопка "Создать курс" ведет сразу на `/teacher/courses/create`, минуя выбор шаблона.

Почему это плохо:

- Один и тот же пользовательский сценарий начинается по-разному.
- Пользователь может вообще не узнать, что существуют шаблоны.
- Поведение кнопок с одинаковым смыслом отличается.

Приоритет: высокий.

### 3. Не объясняется модель "курс -> модуль -> урок -> блок"

В интерфейсе есть модули, уроки и блоки, но нет короткого обучения перед первым использованием.

Почему это плохо:

- Новому преподавателю сложно понять, что такое модуль.
- Непонятно, почему контент нельзя добавить прямо на уровне курса.
- Непонятно, что все материалы урока являются блоками.
- Непонятно, чем "упражнение внутри урока" отличается от "теста".

Приоритет: высокий.

### 4. Наполнение урока спрятано за маленькой иконкой

В редакторе структуры урок редактируется через иконку "Редактировать блоки".

Почему это плохо:

- Основное действие плохо заметно.
- Пользователь может подумать, что переименование урока и есть редактирование урока.
- Нет явной кнопки "Наполнить урок" или "Открыть редактор урока".

Приоритет: средний.

### 5. Тесты и задания создаются вне контекста урока

Блок "Встроенный тест" позволяет выбрать уже существующий тест. Чтобы создать новый тест, пользователь уходит на отдельную страницу `/teacher/test/new`.

Блок "Встроенное задание" работает аналогично: нужно создать задание отдельно, потом выбрать его в блоке.

Почему это плохо:

- Пользователь теряет контекст урока.
- После создания теста/задания непонятно, как вернуться назад и привязать его к нужному блоку.
- Нужно вручную выбирать курс внутри редактора теста/задания, хотя курс уже известен из текущего урока.
- Это делает простое действие "добавить тест в урок" слишком длинным.

Приоритет: высокий.

### 6. Блоки создаются пустыми, но бэк требует заполненные данные

Фронт создает блок сразу после выбора типа. При этом бэк валидирует содержимое уже на создании.

Примеры:

- Текстовый блок требует непустой текст.
- Видео требует URL.
- Аудио требует URL.
- Изображение требует URL.
- Файл требует attachmentId.
- Тест требует выбранный testId.
- Задание требует выбранный assignmentId.
- Упражнения требуют заполненный вопрос и варианты ответов.

Почему это плохо:

- Пользователь нажимает "Добавить блок", но блок может не создаться.
- Пользователь еще не получил форму для заполнения, но система уже требует заполненные данные.
- Это конфликт между UX редактора и серверной валидацией.

Ожидаемое поведение:

- Блок можно создать пустым как черновик.
- Проверка заполненности должна происходить при сохранении блока как готового или при публикации курса.
- В интерфейсе нужно показывать статус блока: "Черновик", "Нужно заполнить", "Готов".

Приоритет: критический.

### 7. Библиотека блоков перегружена для новичка

Сейчас в меню добавления блока есть группы:

- Контент.
- Упражнения - автопроверка.
- Ручная проверка.
- Вставить.

Сами группы полезные, но для новичка список все равно выглядит сложным.

Почему это плохо:

- Непонятно, когда использовать "Пропуски", "Банк слов", "Сопоставление", "Порядок карточек".
- Непонятно, чем "Открытый ответ" отличается от "Встроенного задания".
- Непонятно, когда лучше использовать встроенное упражнение, а когда полноценный тест.

Приоритет: средний.

### 8. Нет обучающих подсказок и повторного запуска инструкции

В редакторе курса и урока нет кнопки с вопросительным знаком, которая объясняет процесс.

Почему это плохо:

- Пользователь должен сам догадаться, как работает редактор.
- Если пользователь вернулся через несколько дней, ему негде быстро вспомнить процесс.
- Нет guided tour для первого запуска.

Ожидаемое поведение:

- При первом входе в редактор курса показывается короткое обучение.
- В правом верхнем углу есть кнопка `?`.
- По кнопке `?` пользователь может повторно открыть инструкцию.
- Инструкция должна быть в формате анимированных карточек/модалок.

Приоритет: высокий.

### 9. Публикационный чеклист появляется слишком поздно

Проверка структуры происходит при публикации.

Почему это плохо:

- Пользователь узнает о проблемах только в конце.
- Не видно заранее, какие уроки пустые.
- Нет постоянного индикатора готовности курса.

Ожидаемое поведение:

- В редакторе курса должен быть постоянный чеклист готовности.
- Ошибки и предупреждения должны отображаться рядом с модулями, уроками и блоками.
- Публикация должна быть финальным подтверждением, а не первым местом, где пользователь видит проблемы.

Приоритет: высокий.

### 10. Автосохранение есть, но его роль не до конца понятна

В редакторе курса и урока есть статус "Автосохранение", "Сохраняем", "Сохранено".

Почему это может быть проблемой:

- Пользователь не всегда понимает, что именно уже сохранено.
- Настройки курса, структура, блоки и порядок блоков сохраняются разными запросами.
- При ошибке сохранения блока сообщение может быть недостаточно связано с конкретным блоком.

Ожидаемое поведение:

- Статус сохранения должен быть виден рядом с конкретным объектом: курс, урок, блок.
- У блока с ошибкой нужно показывать локальную ошибку прямо в карточке блока.

Приоритет: средний.

### 11. Дублирование урока копирует только каркас

В редакторе структуры есть действие "Дублировать урок", но сейчас оно копирует только название/описание/настройки урока, а блоки нужно копировать вручную.

Почему это плохо:

- Пользователь ожидает полную копию урока.
- Текущее поведение может привести к потере времени и ошибкам.

Ожидаемое поведение:

- Либо дублировать урок вместе с блоками.
- Либо явно назвать действие "Дублировать каркас урока".

Приоритет: средний.

### 12. Нет единого "конструктора курса"

Сейчас есть отдельный редактор структуры и отдельный редактор урока.

Почему это плохо:

- Пользователь постоянно переключается между уровнями.
- При наполнении урока не видно всей структуры курса.
- Трудно быстро перейти к следующему уроку.

Ожидаемое поведение:

- Единый Course Builder.
- Слева дерево курса: модули и уроки.
- В центре редактор выбранного урока.
- Справа подсказки, настройки и чеклист.
- Верхняя панель показывает общий прогресс готовности курса.

Приоритет: высокий.

## Рекомендуемая группировка блоков

### Материал

- Текст.
- Видео.
- Аудио.
- Изображение.
- Файл.
- Баннер.

### Быстрая практика

- Один вариант.
- Несколько вариантов.
- Верно / неверно.
- Пропуски.
- Банк слов.
- Порядок карточек.
- Сопоставление.

### Проверка

- Встроенный тест.
- Встроенное задание.
- Открытый ответ.
- Упражнение по коду.

## Минимальный план улучшений

1. Сделать один вход в создание курса: все кнопки "Создать курс" должны вести на выбор шаблона или единый мастер.
2. Добавить onboarding-модалку с объяснением модели "курс -> модуль -> урок -> блок".
3. Добавить кнопку `?` для повторного открытия инструкции.
4. Исправить конфликт создания пустых блоков и серверной валидации.
5. Сделать понятную кнопку "Наполнить урок" вместо одной маленькой иконки.
6. Дать возможность создавать тест или задание прямо из блока урока.
7. Добавить постоянный чеклист готовности курса.
8. Переименовать или исправить дублирование урока, чтобы ожидание совпадало с поведением.
