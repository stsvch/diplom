# Проектирование образовательной платформы (LMS)

## Общее описание проекта

Разработка веб-платформы для организации учебного процесса — универсальной образовательной платформы, объединяющей функции LMS (системы управления обучением) и EdTech-платформы. Система предназначена для создания и прохождения курсов по различным дисциплинам, проведения занятий с преподавателем, контроля знаний и отслеживания прогресса обучения.

Ключевой принцип системы — **гибкая настраиваемость**: преподаватель сам определяет параметры курса, заданий, тестов и оценивания под конкретную дисциплину и формат обучения.

## Этап 1. Анализ аналогов

### Рассмотренные аналоги

#### 1. Moodle
- **Тип:** Open-source, self-hosted LMS.
- **Сильные стороны:** максимально гибкая настройка, курсы с модулями, тесты (много типов вопросов), задания с загрузкой файлов и ручной проверкой, журнал оценок, форумы, чаты, вики, календарь дедлайнов, отслеживание прогресса.
- **Слабые стороны с точки зрения пользователя:** перегруженный устаревший интерфейс, избыточность настроек для простых задач, создание курса требует много шагов, запутанная навигация.

#### 2. Google Classroom
- **Тип:** Облачная, бесплатная.
- **Сильные стороны:** быстрое создание курса, простой интерфейс, публикация заданий и материалов, оценивание, интеграция с Google Docs/Drive/Meet.
- **Слабые стороны:** нет встроенного тестирования (нужен Google Forms), нет модулей/структурирования курса (всё лентой), нет отслеживания прогресса, нет полноценного журнала, привязка к экосистеме Google.

#### 3. Canvas LMS
- **Тип:** Облачная / self-hosted.
- **Сильные стороны:** современный интерфейс, курсы с модулями, встроенные тесты с автопроверкой, журнал оценок с гибкими весами, календарь, мобильное приложение.
- **Слабые стороны:** ориентирован на крупные вузы, многие функции только в платной версии, интерфейс создания курса требует обучения.

#### 4. Stepik
- **Тип:** Облачная, ориентация на онлайн-курсы.
- **Сильные стороны:** пошаговое прохождение, автоматическая проверка (тесты, задачи с кодом), простой интерфейс для студента, прогресс-бар, комментарии и обсуждения.
- **Слабые стороны:** слабая поддержка ручной проверки заданий, нет полноценного журнала успеваемости, ограниченные возможности для управления группой, заточен под MOOC, а не под учебное заведение.

#### 5. Microsoft Teams for Education
- **Тип:** Облачная, средство коммуникации с элементами LMS.
- **Сильные стороны:** задания с дедлайнами и оценками, интеграция с Office 365, видеозвонки, чаты, загрузка файлов.
- **Слабые стороны:** не LMS по сути, нет структурирования курса, нет тестов с автопроверкой (нужен Forms), нет прогресса прохождения, нет полноценного журнала.

### Сводная таблица сравнения (с точки зрения пользователя)

| Функция | Moodle | Google Classroom | Canvas | Stepik | MS Teams |
|---------|--------|-----------------|--------|--------|----------|
| Структурирование курса (модули) | + | - | + | + | - |
| Тесты с автопроверкой | + | - | + | + | - |
| Задания с ручной проверкой | + | + | + | - | + |
| Журнал оценок | + | частично | + | - | частично |
| Прогресс прохождения | + | - | + | + | - |
| Простота интерфейса | - | + | частично | + | частично |
| Уведомления | + | + | + | + | + |
| Быстрота создания курса | - | + | - | частично | - |

### Выявленные пробелы аналогов

1. Moodle / Canvas — мощные, но сложные в использовании для рядового пользователя.
2. Google Classroom / MS Teams — простые, но не имеют полноценной LMS-функциональности (нет модулей, тестов, прогресса).
3. Stepik — хорош для автопроверки, но слаб в ручной проверке и управлении группой.
4. Ни один аналог не совмещает одновременно: простоту интерфейса + полный набор LMS-функций + оба типа проверки (авто + ручная) + поддержку разных дисциплин + занятия с преподавателем + монетизацию.

### Ключевое пользовательское преимущество проектируемой системы

Система предоставляет **гибкость Moodle с простотой Google Classroom**: преподаватель при создании курса/задания/теста сам определяет все параметры (платность, оценки, дедлайны, попытки, тип проверки), а интерфейс остаётся понятным и не перегруженным.

## Этап 2. Функциональные требования

### Роли пользователей

В системе предусмотрены три роли:
- **Администратор** — управление платформой, пользователями, модерация.
- **Преподаватель** — создание курсов, заданий, тестов, оценивание, проведение занятий.
- **Студент** — прохождение курсов, выполнение заданий, сдача тестов.

**Гость** (незарегистрированный пользователь) — может просматривать каталог курсов, но для работы с платформой обязана пройти регистрацию.

### Зафиксированные решения по настраиваемости

Система должна быть гибко настраиваемой. Преподаватель при создании курса / задания / теста сам определяет:

| Параметр | Варианты |
|----------|----------|
| Тип курса | Платный / бесплатный |
| Оценивание | С оценками / без оценок |
| Порядок прохождения | Последовательный / свободный |
| Дедлайн | С дедлайном / без дедлайна |
| Количество попыток | 1 / N / без ограничений |
| Тип проверки | Автоматическая / ручная |
| Возврат на доработку | Включён / выключен |

---

### 1. Регистрация, авторизация, роли

- Система должна обеспечивать регистрацию пользователей (email + пароль).
- Система должна обеспечивать вход и выход из системы.
- Система должна поддерживать три роли: администратор, преподаватель, студент.
- Гость (незарегистрированный пользователь) может просматривать каталог курсов.
- Система должна разграничивать доступ к функциям в зависимости от роли.
- Система должна предоставлять возможность восстановления пароля.

**Перспективное:** вход через социальные сети (Google, VK) — планируется в будущем, в текущей версии только email.

### 2. Курсы и контент

- Система должна предоставлять преподавателю возможность создавать курсы.
- Курс должен иметь иерархическую структуру: курс → модули → уроки.
- Система должна поддерживать платные и бесплатные курсы.
- Система должна обеспечивать размещение в уроке следующих типов контента:
  - текстовые материалы;
  - файлы (PDF, документы, презентации);
  - видеоматериалы (загрузка файла или ссылка на внешний ресурс);
  - ссылки на внешние ресурсы.
- Система должна позволять преподавателю публиковать, скрывать и архивировать курсы.
- Система должна поддерживать настраиваемый порядок прохождения курса:
  - последовательный (урок N+1 доступен только после завершения урока N);
  - свободный (все уроки доступны сразу).
- Система должна обеспечивать каталог курсов с поиском и фильтрацией.
- Система должна обеспечивать студенту доступ к содержимому назначенных/приобретённых курсов.

### 3. Тесты и проверка знаний

- Система должна предоставлять преподавателю возможность создавать тесты вручную.
- Система должна поддерживать следующие типы вопросов:
  - выбор одного правильного ответа;
  - выбор нескольких правильных ответов;
  - ввод текстового ответа;
  - сопоставление (установление соответствия);
  - открытый вопрос (с ручной проверкой преподавателем).
- Система должна обеспечивать автоматическую проверку закрытых вопросов.
- Система должна обеспечивать ручную проверку открытых вопросов преподавателем.
- Система должна поддерживать настройки теста:
  - ограничение количества попыток (1 / N / без ограничений);
  - таймер на выполнение;
  - дедлайн сдачи (опционально);
  - скрытие правильных ответов до завершения;
  - перемешивание вопросов и вариантов ответов.
- Система должна сохранять результаты каждой попытки прохождения.

**Перспективное:** автоматическая генерация тестов на основе учебного материала — механизм реализации будет определён позднее.

### 4. Задания

- Система должна позволять преподавателю создавать задания с описанием и критериями оценивания.
- Система должна поддерживать настраиваемый дедлайн (опционально).
- Система должна обеспечивать загрузку ответа студентом (файл, текст, код).
- Система должна обеспечивать проверку задания преподавателем.
- Система должна поддерживать выставление оценки и написание комментария.
- Система должна поддерживать настраиваемое количество попыток сдачи (1 / N / без ограничений).
- Система должна поддерживать возврат задания на доработку (если попытки не исчерпаны).
- Система должна отображать статус задания:
  - не начато → отправлено → на проверке → оценено → возвращено на доработку.

### 5. Оценивание и журнал

- Система должна обеспечивать автоматическое выставление баллов за тесты.
- Система должна обеспечивать ручное выставление оценок преподавателем за задания.
- Система должна поддерживать настраиваемый режим оценивания: курс с оценками или без оценок.
- Система должна вести электронный журнал успеваемости.
- Система должна формировать итоговый балл по модулю и по курсу.
- Система должна обеспечивать экспорт журнала (PDF / Excel).

### 6. Занятия с преподавателем

- Преподаватель должен иметь возможность создавать слоты для индивидуальных и групповых занятий.
- Студент должен иметь возможность записаться на занятие.
- Система должна поддерживать статусы занятия: запланировано → проведено → отменено.
- Система должна вести историю проведённых занятий.
- Система должна отправлять напоминания о предстоящих занятиях.

**Требует дополнительного обсуждения:** формат проведения занятий (встроенный видеозвонок или внешняя ссылка), детали реализации.

### 7. Подписки и оплата

- Система должна поддерживать бесплатные курсы.
- Система должна поддерживать платные курсы (разовая покупка).
- Система должна поддерживать подписки (доступ к набору курсов на период).
- Система должна поддерживать пакеты занятий с преподавателем.
- Система должна обеспечивать интеграцию с платёжной системой.
- Система должна вести историю платежей.

### 8. Коммуникация

- Система должна обеспечивать чат внутри курса (между студентами и преподавателем).
- Система должна поддерживать личные сообщения между пользователями.
- Система должна поддерживать комментарии к заданиям и урокам.
- Система должна обеспечивать уведомления о следующих событиях:
  - публикация нового задания или теста;
  - приближение дедлайна;
  - оценка выставлена;
  - задание возвращено на доработку;
  - новое занятие / напоминание о занятии;
  - публикация нового материала в курсе.

### 9. Прогресс и аналитика

- Система должна отслеживать прохождение уроков (отметка о просмотре/завершении).
- Система должна отображать процент завершённости курса.
- Система должна предоставлять студенту личный дашборд: мои курсы, прогресс, оценки, ближайшие дедлайны.
- Система должна предоставлять преподавателю дашборд: статистика по курсу, кто прошёл, кто отстаёт.
- Система должна формировать отчёты по успеваемости (по студенту, по курсу, по группе).

### 10. Инструменты по дисциплинам

Система должна поддерживать специализированные инструменты для различных дисциплин:

- **Языковые курсы:**
  - словарь терминов/слов с возможностью пополнения;
  - упражнения на заучивание (карточки, подстановка);
  - автопроверка знания слов.
- **Программирование:**
  - встроенный редактор кода;
  - автоматическая проверка кода (запуск и сравнение с эталонным результатом).
- **Универсальные инструменты:**
  - тесты, задания, материалы — применимы ко всем дисциплинам.

Система должна поддерживать максимально широкий набор дисциплин, используя комбинацию универсальных и специализированных инструментов.

### 11. Календарь

- Система должна отображать календарь с дедлайнами заданий и тестов.
- Система должна отображать запланированные занятия с преподавателем.
- Система должна отправлять напоминания о приближающихся событиях.

### 12. Администрирование

- Администратор должен иметь возможность управлять пользователями (создание, блокировка, назначение ролей).
- Администратор должен иметь возможность управлять курсами (модерация, удаление).
- Администратор должен иметь доступ к общей аналитике платформы.
- Администратор должен иметь возможность управлять настройками платформы.

## Входные данные системы

- Регистрационные данные пользователей (email, пароль, имя, роль).
- Сведения о курсах (название, описание, категория, настройки, стоимость).
- Данные о модулях и уроках (структура, порядок, контент).
- Учебные материалы (тексты, файлы, видео, ссылки).
- Параметры заданий (название, описание, дедлайн, критерии, число попыток).
- Параметры тестов (вопросы, варианты ответов, правильные ответы, настройки).
- Файлы, загружаемые преподавателем или студентом.
- Результаты выполнения заданий (ответы студентов).
- Оценки и комментарии преподавателя.
- Данные словаря (слова, переводы, определения).
- Код, написанный студентом (для курсов по программированию).
- Расписание занятий (слоты, время, формат).
- Платёжные данные (при покупке курсов или подписок).

## Выходные данные системы

- Каталог курсов с фильтрацией и поиском.
- Структура курса (модули, уроки, материалы).
- Отображение учебных материалов (текст, файлы, видео).
- Перечень заданий и тестов с их статусами.
- Результаты прохождения тестов (баллы, детализация ответов).
- Результаты проверки заданий (оценка, комментарий).
- Электронный журнал успеваемости.
- Итоговый балл по модулю и курсу.
- Прогресс прохождения курса (процент завершённости).
- Дашборды студента и преподавателя.
- Отчёты по успеваемости (по студенту, курсу, группе).
- Календарь дедлайнов и занятий.
- Уведомления о событиях.
- История платежей и статус подписок.
- Результаты упражнений по словарю / проверке кода.

## Этап 3. Нефункциональные требования

### 1. Производительность и масштабируемость

- Архитектура системы должна обеспечивать возможность горизонтального масштабирования при росте нагрузки.
- Система должна быть спроектирована таким образом, чтобы в перспективе обеспечивать переход от модульного монолита к микросервисной архитектуре без существенной переработки бизнес-логики.
- Время отклика пользовательского интерфейса не должно превышать 2 секунд для основных операций.

### 2. Безопасность

- Система должна обеспечивать хэширование паролей (хранение в открытом виде недопустимо).
- Система должна использовать токены (JWT) для аутентификации и управления сессиями.
- Система должна обеспечивать защиту от основных веб-уязвимостей (XSS, CSRF, SQL-инъекции).
- Передача данных должна осуществляться по протоколу HTTPS.
- Система должна обеспечивать строгое разграничение доступа по ролям.
- Все входные данные должны проходить валидацию.

**Перспективное:** двухфакторная аутентификация (2FA) — планируется в будущих версиях.

### 3. Надёжность

- Система должна сохранять введённые данные при сбоях (ответы на тесты, загруженные файлы).
- Система должна поддерживать автосохранение прогресса при прохождении тестов.
- Должно быть предусмотрено резервное копирование базы данных.

### 4. Требования к интерфейсу

- Интерфейс должен быть адаптивным (корректное отображение на десктопе, планшете, мобильных устройствах).
- Язык интерфейса — русский.
- Интерфейс должен быть простым и интуитивно понятным, с минимальным количеством шагов для выполнения основных операций.

### 5. Совместимость

- Система должна корректно работать во всех современных браузерах (Chrome, Firefox, Safari, Edge).
- Мобильное приложение не предусмотрено — используется адаптивная веб-версия.

### 2. Архитектурный подход

#### Общий подход — модульный монолит

Серверная часть строится как модульный монолит: единое приложение, но с чётким разделением на логические модули. Это позволяет в перспективе при необходимости выделить отдельные модули в микросервисы.

#### Архитектура внутри модулей — Onion Architecture (луковая архитектура)

Каждый модуль строится по принципам Onion Architecture с разделением на слои:

```
Domain (Entities, Value Objects)       — ядро, бизнес-сущности и правила
    ↑
Application (Services, CQRS)          — бизнес-логика, команды и запросы
    ↑
Infrastructure (Repositories, EF)     — доступ к данным, внешние сервисы
    ↑
API (Controllers, DTOs, Middleware)    — входная точка, HTTP-контракты
```

Зависимости направлены внутрь: внешние слои зависят от внутренних, но не наоборот. Domain-слой не имеет зависимостей от инфраструктуры.

#### CQRS (Command Query Responsibility Segregation)

Для разделения операций чтения и записи применяется паттерн CQRS:
- **Commands** — операции, изменяющие состояние системы (создание курса, отправка задания, выставление оценки).
- **Queries** — операции чтения данных (получение списка курсов, просмотр журнала, отчёты).

Разделение позволяет оптимизировать модели чтения и записи независимо друг от друга.

## Этап 5. Модель данных

### Принципы разделения хранилищ

| Хранилище | Что хранит |
|-----------|-----------|
| **PostgreSQL** | Пользователи, дисциплины, курсы, модули, уроки, тесты, задания, оценки, прогресс, подписки, платежи, уведомления, комментарии, метаданные вложений |
| **MongoDB** | Чаты, сообщения (включая вложения к сообщениям) — высоконагруженные данные с гибкой структурой |
| **S3 / MinIO** | Физические файлы: документы, видео, изображения, ответы студентов |

**Обоснование вынесения чатов в MongoDB:** сообщения — это высоконагруженные данные с быстрым ростом объёма, гибкой структурой (вложения, статусы прочтения, реакции). Хранение в реляционной БД приводит к избыточным JOIN-ам и снижению производительности при масштабировании.

### Связи между сущностями

```
Discipline (1) ──── (N) Course                  [дисциплина → курсы]
Discipline (1) ──── (N) ScheduleSlot            [дисциплина → слоты занятий]
Discipline (1) ──── (N) DictionaryWord          [дисциплина → словарь]
User (1) ──── (N) Course                        [преподаватель → его курсы]
User (1) ──── (N) CourseEnrollment              [студент → записи на курсы]
Course (1) ──── (N) CourseEnrollment            [курс → записанные студенты]
Course (1) ──── (N) CourseModule                [курс → модули]
CourseModule (1) ──── (N) Lesson                [модуль → уроки]
Lesson (1) ──── (N) LessonBlock                  [урок → блоки контента]
LessonBlock (0..1) ──── (1) Test                 [блок type=Quiz → тест]
LessonBlock (0..1) ──── (1) Assignment           [блок type=Assignment → задание]
LessonBlock (0..1) ──── (1) Exercise             [блок type=Exercise → упражнение]
Test (1) ──── (N) Question                      [тест → вопросы]
Question (1) ──── (N) AnswerOption               [вопрос → варианты ответов]
Test (1) ──── (N) TestAttempt                   [тест → попытки прохождения]
TestAttempt (1) ──── (N) TestResponse           [попытка → ответы на вопросы]
Assignment (1) ──── (N) AssignmentSubmission     [задание → сдачи работ]
User (1) ──── (N) Grade                         [студент → оценки в журнале]
Course (1) ──── (N) Grade                       [курс → оценки]
User (1) ──── (N) LessonProgress                [студент → прогресс по урокам]
User (1) ──── (N) ScheduleSlot                  [преподаватель → слоты]
ScheduleSlot (1) ──── (N) SessionBooking        [слот → записи студентов]
User (1) ──── (N) Payment                       [пользователь → платежи]
User (1) ──── (N) UserSubscription              [пользователь → подписки]
User (1) ──── (N) Notification                  [пользователь → уведомления]
Exercise (1) ──── (N) ExerciseSubmission         [упражнение → отправки ответов]
Attachment (N) ──── (1) любая сущность           [полиморфная связь: EntityType + EntityId]
```

**Итого:** 27 сущностей в PostgreSQL (LessonContent → LessonBlock, CodeExercise → Exercise, CodeSubmission → ExerciseSubmission), 2 коллекции в MongoDB, файлы в S3/MinIO.

### Изменения в модели данных (обновлено)

1. **LessonContent заменён на LessonBlock** — урок теперь контейнер из упорядоченных блоков разных типов (Text, Video, File, Quiz, Assignment, Exercise). Один урок может содержать видео + текст + тест + задание одновременно.
2. **CodeExercise заменён на Exercise** — универсальная сущность для упражнений разных типов (Code, Vocabulary, FillBlank, Matching, Ordering), не только для кода.
3. **CodeSubmission заменён на ExerciseSubmission** — универсальная отправка ответа на любой тип упражнения.

## Известные ограничения и технический долг

### Функциональные gaps (требуют реализации)

1. **Календарь не агрегирует дедлайны из Assignments/Tests** — CalendarEvent хранит только вручную созданные события. Автоматическое создание событий при задании дедлайна в Assignment/Test не реализовано. Нужно добавить при создании Assignment с Deadline / Test с Deadline → автоматически создавать CalendarEvent.

2. **Gradebook показывает userId вместо имён студентов** — модуль Grading изолирован от Auth, поэтому не имеет доступа к именам пользователей. Для решения нужно: либо enrichment на уровне контроллера (запрос к UserManager), либо хранение имён в Grade entity (денормализация).

3. **GetMyProgress пропускает курсы с 0 завершённых уроков** — Progress endpoint находит курсы только через завершённые уроки. Студенты, записанные на курс, но не начавшие обучение, не видят курс в прогрессе. Нужно: получать список записанных курсов из CourseEnrollment.

4. **Пустой gradebook экспорт** — при отсутствии данных экспорт генерирует пустой файл без информативного сообщения. Нужно: возвращать 404 или ошибку.

### Перспективные вопросы

5. **Занятия с преподавателем** — формат проведения (встроенный видеозвонок или внешняя ссылка), подробности реализации.
6. **Автогенерация тестов** — механизм реализации (AI на основе материала или банк вопросов), детали будут определены позднее.
3. **Дисциплины** — окончательный перечень специализированных инструментов помимо языков и программирования.
